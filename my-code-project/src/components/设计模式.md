# 设计思想与原则
  #SOLID
    -   单一职责原则（Single Responsibility Principle，SRP）
    -   开放/封闭原则（Open/Closed Principle，OCP）
    -   里氏替换原则（Liskov Substitution Principle，LSP）
    -   接口隔离原则（Interface Segregation Principle，ISP）
    -   依赖反转原则（Dependency Inversion Principle，DIP）

# 1、创建型
    -   工厂方法模式
        - 它定义了一个用于创建对象的接口，但是由子类决定实例化哪个类。这样，工厂方法模式将类的实例化推迟到子类

    -   抽象工厂模式
        - 它创建了一种接口，用于创建一系列相关或相互依赖的对象，而无需指定它们的具体类。
          抽象工厂模式是工厂方法模式的进一步扩展，它将一组组的工厂封装在一个接口中，
          使客户端代码可以与具体工厂的实现分离

    -   原型模式
        - 通过复制现有对象来创建新对象，而无需显示的使用构造函数。原型模式通常用于创建复杂的对象，
          而新对象的创建可以通过复制现有对象的状态来更高效的实现

    -   单例模式
        - 其主要目的是确保一个类只有一个实例，并提供一个全局访问点以获取该实例

# 2、结构型
    - 适配器模式
        -它允许将一个类的接口转换成客户端期望的另外一个接口，适配器模式主要用于使原本由于接口不匹配而无法一起工作的
        两个类能够协同工作

    - 桥接模式
        - 它将抽象部分和实现部分分离，使它们可以独立变化，而不影响彼此。桥接模式主要用于处理具有多层次继承结构的
        情况，通过将抽象部分和实现部分分开，使得系统更加灵活

    - 组合模式
        - 它允许你将对象组合成树形结构以表示"部分-整体"的层次结构。组合模式使得客户端对单个对象和组合对象的使用
        具有一致性

    - 装饰者模式
        - 它允许你通过将对象放入放入包含行为的特殊包装类中来给对象动态地添加新地行为。装饰者模式提供了一种灵活地方式
        ，使得客户端可以单独扩展对象地功能，而不需要修改其结构

    - 外观模式
        - 提供了一个统一的接口，用于访问子系统中的一群接口。外观模式定义了一个高层接口，使得子系统更容易使用

    - 享元模式
        - 通过共享对象来减少内存和计算开销，该模式适用于系统中存在大量相似对象，通过共享这些对象的相同部分，
        减少了实例的数量，从而降低了内存占用和提高了性能

    - 代理模式
        - 其主要目的是通过引入一个代理对象来控制另一个对象的访问。代理对象充当了客户端和目标对象中间的中介，
        可以用于实现各种用途，如 延迟加载，权限控制，日志记录等

# 3、行为型
    - 模板方法模式
        - 定义了一个操作中的算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以不改变一个算法的结构即可
        重定义该算法的某些特定步骤。

    - 观察者模式
        - 定义对象间的一种一对多依赖关系，使得每当一个对象状态发生改变时，其相关依赖对象皆能得到通知并自动更新
        其实就是为了实现 松耦合

    - 状态模式
        - 允许一个对象在其内部状态改变时改变它的行为，对象看起来似乎修改了它的类。其别名为状态对象（Objects for States）
        状态模式是一种对象行为型模式

    - 策略模式
        - 定义一系列算法，将每一个算法封装起来，并让它们可以相互替换。策略模式让算法独立于使用它的客户而变化，也称为
        政策模式（policy）

    - 职责链模式
        - 解决请求的发送者和请求的接收者之间的耦合，通过职责链上的多个对象对分解请求流程，实现请求在多个对象之间的传递，
        直到有对象处理它为止

    - 命令模式
        - 将一个请求封装为一个对象，从而使我们可用不同的请求对客户进行参数化；对请求排队或者记录请求日志。以及支持可
        撤销的操作。命令模式是一种对象行为型模式，别名为动作（Action）模式或事务（Transaction）模式

    - 访问者模式

    - 迭代器模式
    - 解释器模式
